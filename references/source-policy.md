# Source Policy

每次预测都要先查证，再判断。足球赛前信息高度易变，尤其是名单、伤停、停赛、赛前发布会、赛地环境和更衣室动态。

同时要控制搜索成本。这个 skill 默认不是深度情报调查，而是赛前快速起盘。

## 搜索预算

默认使用快筛模式：

- 最多两轮搜索。
- 目标来源数：4-6 个。4 个高价值来源已经填满证据矩阵时即可停止。
- 最少可接受来源数：3 个，但必须包含一个官方/权威赛程来源、一个名单或可用阵容来源、一个伤停/状态/环境相关来源，并清楚列出缺口。
- 不要为了凑够来源数量继续搜索；不要把二级转载当作"补充严谨性"。
- 不要为了证明"没有伤病"或"没有更衣室矛盾"无限搜索；完成可靠源快筛后，找不到就写"未找到可靠公开来源"。
- 不抓取、不分析、不输出任何竞猜交易类页面内容；如果用户要求这类内容，说明本 skill 只处理正规赛前信息和娱乐比分预测。
- 同一场比赛在同一轮对话内复用已核验的赛程、地点、历史交锋和名单；用户要求刷新或临近开赛时，只更新伤停、首发、赛前发布会和赛地环境。

## 时间预算

- 第一段赛前情报目标耗时：2-4 分钟。
- 第一段硬停止线：5 分钟。到线仍未填满证据矩阵时，停止搜索并把未填项写入"情报缺口"。
- 单个网页或搜索结果明显卡住时，不要等待整轮超时；换一个同级可靠来源，或记录缺口。
- 如果 agent 支持并行搜索/浏览，赛程、名单、伤停、状态/交锋和环境变量来源应并行抓取，再合并去重。
- 如果 agent 只能顺序浏览，严格按推荐搜索顺序执行；每个格子只保留第一个足够可靠的来源。

## 证据矩阵和停止条件

把搜索当作填表，而不是泛读网页。每个格子满足后就不要重复查同类信息：

| 格子 | 满足条件 | 停止条件 |
|---|---|---|
| 赛程 | 官方 Match Centre、赛事官网、足协/俱乐部官网，或主流数据源锁定比赛身份 | 日期、地点、赛事阶段清楚后停止 |
| 名单 | 官方名单、赛事名单页、国家队/俱乐部公告，或权威媒体名单页 | 最新可用名单状态清楚后停止 |
| 伤停/停赛 | 官方发布会、主流 team news、可靠伤停数据源或权威记者 | 有关键缺席/复出，或可靠源快筛后未找到 |
| 更衣室动态 | 官方发布、主流媒体署名报道、可靠跟队记者或多源交叉确认 | 只查可靠负面信息；未找到就停止，不证明不存在 |
| 状态/交锋 | 主流数据页、赛事预览、Opta/Stats Perform 类数据、11v11/WorldFootball 等 | 得到近期趋势和交锋结论后停止 |
| 环境/变量 | 官方赛地信息、权威预览、天气/海拔/旅行/赛程密度等正规来源 | 得到真正影响比赛的外部变量，或确认没有显著变量后停止 |

## 推荐搜索顺序

按这个顺序查，信息够了就停：

1. 官方赛程/Match Centre：FIFA、赛事官网、足协或官方比赛页。
2. 官方或权威名单：赛事名单页、国家队官网、FourFourTwo/ESPN/BBC 等名单页。
3. 伤停/停赛和团队动态：主流媒体、官方发布会、可靠跟队记者。更衣室只查可靠负面信息，不做泛搜。
4. 状态/交锋：主流数据页、赛事预览、历史交锋页。
5. 环境/变量：官方赛地信息、天气、海拔、旅行距离、赛程密度、权威预览里的战术或体能变量。
6. 本地/主流媒体补充：只补关键变量，不做泛读。

## 来源优先级

从高到低：

1. 官方来源：FIFA、洲际足联、赛事官网、国家队/俱乐部官网、官方社媒、官方比赛报告。
2. 主流数据和新闻：ESPN、BBC、The Guardian、Reuters、AP、Sky Sports、Opta/Stats Perform、The Athletic、主流当地体育媒体。
3. 可靠署名记者：长期跟队、能被主流媒体引用、有稳定记录的记者。
4. 数据库：11v11、WorldFootball、Transfermarkt、FBref、SofaScore、FotMob 等，用于历史交锋、出场、伤停、近期状态。
5. 环境来源：官方场馆页、赛事城市页、主流天气服务、权威预览中的赛地/旅行/赛程信息。
6. Wikipedia：只能兜底查赛程、历史交锋、名单汇总等稳定信息；不能标为高可信，不能用来证明"没有伤病"或"没有更衣室问题"。

## 推荐查询形态

优先用单次查询覆盖多个格子，避免每个问题单独开新搜索：

- `{team A} {team B} match centre lineups injury venue`
- `{team A} {team B} preview team news injuries venue`
- `{team A} squad injury team news {competition}`
- `{team B} squad injury team news {competition}`

如果官方来源已经锁定赛程，不要再搜索多个赛程页。如果权威预览已经同时覆盖伤停、状态和环境变量，不要把这三类拆成三轮搜索。

## 必须覆盖的事实

每场比赛至少尝试查证，并压缩成 6 行核心情报：

- 赛程：赛事、日期、地点、阶段。
- 名单：最终名单、临时名单、最新可用球员，或官方名单尚未发布。
- 伤停：关键缺席、待定、停赛、复出，或未找到可靠公开来源。
- 更衣室：可靠团队动态；没有可靠来源就写没有找到。
- 状态/交锋：近期状态和历史交锋合并成一句，不展开长列表。
- 环境/变量：主客场、旅行、海拔、天气、场地、赛程密度或权威预览提到的关键外部变量。

不要把每个网页都写成一段摘要。输出给用户的赛前情报每行只保留一个最影响预测的事实。

## 缓存和刷新

同一场比赛在同一轮对话内，默认复用这些已经核验的信息：

- 赛程、地点、赛事阶段。
- 历史交锋和长期数据。
- 已公布的大名单或官方可用名单。

默认只刷新这些易变信息：

- 临场首发。
- 伤停、停赛、复出和赛前发布会。
- 赛地环境、旅行和天气等临场可能变化的外部变量。

如果用户明确要求"重新查一遍"或"最新"，再重新跑完整证据矩阵，但仍遵守 4-6 个来源和停止条件。

## 可信度标签

给每条关键信息标注可信度：

- 高：官方发布、主流媒体署名报道、可靠数据源。
- 中：可靠记者单源消息、专业数据库但未见官方确认。
- 低：社媒传言、小号搬运、没有署名或没有原始来源。
- 兜底：Wikipedia 或其他二级汇总，仅用于辅助，不用于排除伤病/更衣室问题。

低可信信息不得进入最终预测权重。可以写在"排除信息"中，说明为什么不用。

## 禁止事项

- 不要凭记忆给"最新"名单、伤停、赛前发布会或赛地环境。
- 不要用十几个网页堆砌来制造"很严谨"的感觉。
- 不要把访问失败的网站列成正式来源；只在缺口里说明尝试失败。
- 不要把 Wikipedia 标成高可信。
- 不要把模型推测写成新闻事实。
- 不要编造专家原话、内部消息、天气、赛地信息或球队动态。
- 不要抓取、分析或输出竞猜交易类页面。
- 不要用"据传"支撑比分判断。
- 不要输出下注金额、倍投、梭哈、套利、稳赚等建议。
